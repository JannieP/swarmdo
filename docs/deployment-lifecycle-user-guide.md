# Deployment & Lifecycle — User Guide

Commands for the **operational lifecycle** of a Swarmdo project: inspecting **deployments** and **environments**, tracking **V2→V3 migration** state, listing **workflows**, and generating **shell completions**.

The `status` / `list` / `environments` inspectors and every `--help` screen are **read-only** and safe to run anytime. The state-changing commands (`deploy`, `rollback`, `release`, `migrate run`, `workflow run`) **mutate state** — they are documented below from their `--help` text and were **not executed** for this guide. Output shown is from real runs on this repo (a clean checkout with nothing deployed yet), so most inspectors report an honest empty state.

---

## `deployment status` — is anything deployed?

Check deployment status across all configured environments (or one, with `-e`).

```
$ swarmdo deployment status
Deployment Status

No active deployment
No environments configured
```

On a fresh repo this is the expected empty state — no environments have been registered and nothing has shipped. Once environments exist, this reports the active version per environment.

**Use it for:** a quick "what's live where" check before releasing or rolling back.

---

## `deployment environments` — what targets exist?

List (default), add, or remove deployment environments. `--action add -n <name> -t <type>` registers one; `--action remove -n <name>` deletes it.

```
$ swarmdo deployment environments
Deployment Environments

No environments configured. Use --action add to create one.
```

The command tells you exactly how to fix the empty state. To register a target (mutates config — not run here):

```
$ swarmdo deployment envs -a add -n preview -t staging
```

**Use it for:** setting up dev / staging / prod targets before your first deploy.

---

## `migrate status` — do I need to migrate V2→V3?

Report the migration state of each component (Config, Memory, Sessions) and whether a V2→V3 migration is required. Read-only.

```
$ swarmdo migrate status
Migration Status

  Component   Status    Migration Needed
  Config      v2 + v3   no
  Memory      missing   no
  Sessions    missing   no

[OK] No migration needed.
```

Here Config already carries both v2 and v3 keys and no migration is pending. `missing` simply means that component has no legacy data to carry over.

**Use it for:** confirming an upgrade is clean before/after bumping Swarmdo; a pre-flight gate for `migrate run`.

---

## `workflow list` — what workflows are defined?

List saved workflow definitions and templates. Read-only.

```
$ swarmdo workflow list
Workflows

[INFO] No workflows found
```

Empty on a fresh repo. Once you define workflows (via `workflow template` / a YAML file), they appear here with their status.

**Use it for:** discovering available workflows before running or validating one.

## `completions bash` — shell tab-completion

Emit a bash completion script to stdout. Install with `swarmdo completions bash > ~/.bash_completion.d/swarmdo`. Also available: `zsh`, `fish`, `powershell`.

```
$ swarmdo completions bash
# swarmdo bash completion
# Generated by swarmdo completions bash
# Add this to ~/.bashrc or ~/.bash_completion

_claude_flow_completions() {
    local cur prev words cword
    _init_completion -n : || return

    local commands="swarm agent task session config memory workflow hive-mind hooks daemon neural security performance providers plugins deployment claims embeddings doctor completions statusline help version"
    ...
complete -F _claude_flow_completions swarmdo
complete -F _claude_flow_completions npx\ @swarmdo/cli@v3alpha
```

The script works and registers to the `swarmdo` command, but the generated output carries three stale artifacts (see Troubleshooting): the internal function is still named `_claude_flow_completions` (upstream branding), the top-level command list omits several current commands (`init`, `start`, `status`, `mcp`, `migrate`, `usage`, and most newer utility commands), and the second registration targets a nonexistent `@swarmdo/cli@v3alpha` dist-tag.

**Use it for:** tab-completing commands and subcommands in your shell. Prefer `zsh`/`fish` if you're on those shells.

## Lifecycle commands (state-changing — documented from `--help`, not run)

These mutate deployments, config, or spawn workers. Descriptions and options are taken from their `--help` screens; none were executed for this guide.

### `deployment deploy`
Deploy to a target environment.

```
$ swarmdo deployment deploy --help
OPTIONS:
  -e, --env                 Environment: dev, staging, prod [default: staging]
  -v, --version             Version to deploy
  -d, --dry-run             Simulate deployment without changes
      --description         Deployment description
```

**Use it for:** shipping a version to an environment. Use `--dry-run` first to simulate.

### `deployment rollback`
Roll back to a previous deployment.

```
$ swarmdo deployment rollback --help
OPTIONS:
  -e, --env                 Environment to rollback (required)
  -v, --version             Specific version to rollback to
  -s, --steps               Number of versions to rollback [default: 1]
```

**Use it for:** reverting a bad release — one step back by default, or to a pinned `--version`.

### `deployment release`
Create a new release deployment.

```
$ swarmdo deployment release --help
OPTIONS:
  -v, --version             Release version
  -e, --env                 Target environment [default: production]
  -d, --description         Release description
```

**Use it for:** cutting a versioned release (defaults to the `production` environment).

### `migrate run`
Run the V2→V3 migration.

```
$ swarmdo migrate run --help
OPTIONS:
  -t, --target              Migration target
      --dry-run             Show what would be migrated without making changes [default: false]
      --backup              Create backup before migration [default: true]
  -f, --force               Force migration (overwrite existing) [default: false]
```

**Use it for:** carrying legacy V2 data forward. Run `migrate status` first; preview with `--dry-run` (backup is on by default).

### `workflow run`
Execute a workflow (spawns agents).

```
$ swarmdo workflow run --help
OPTIONS:
  -t, --template            Workflow template
  -f, --file                Workflow definition file (YAML/JSON)
      --task                Task description
  -p, --parallel            Enable parallel execution [default: true]
  -m, --max-agents          Maximum agents to spawn [default: 5]
      --timeout             Workflow timeout in minutes [default: 30]
  -d, --dry-run             Validate without executing [default: false]
```

**Use it for:** running a development/SPARC/custom workflow. Validate first with `-d, --dry-run` or `workflow validate`.

> Note: `workflow execute` resolves to the same `workflow` command surface — the canonical run subcommand is `workflow run`.

### `init` / `start`
`init` scaffolds Swarmdo into the current directory (presets, skills, hooks); `start` boots the orchestration system (MCP server, daemon). Both change on-disk state or start processes — see `swarmdo init --help` and `swarmdo start --help`.

## Workflows

| Question | Command(s) |
|----------|-----------|
| What's live, and where? | `deployment status` (add `-e prod` for one env) |
| What deploy targets exist? | `deployment environments` (`-a add`/`-a remove` to change) |
| Do I need to migrate after an upgrade? | `migrate status` → then `migrate run --dry-run` |
| What workflows can I run? | `workflow list` → `workflow validate` → `workflow run` |
| Ship / revert a version | `deployment deploy` / `deployment rollback` (`--dry-run` to preview) |
| Enable shell tab-completion | `completions bash` (or `zsh`/`fish`/`powershell`) |

## Troubleshooting

| Symptom | Explanation / fix |
|---------|-------------------|
| `deployment status` → "No environments configured" | Expected on a fresh repo. Register a target: `deployment environments -a add -n <name> -t <type>`. |
| `migrate status` shows components as `missing` | Not an error — that component has no legacy V2 data to migrate. `Migration Needed: no` = you're clean. |
| `workflow list` → "No workflows found" | No workflows defined yet. Create one via `workflow template` or a YAML file, then it lists here. |
| `completions bash` function is named `_claude_flow_completions` | Stale upstream (claude-flow) branding in generated output. Harmless (internal name); completions still register to `swarmdo`. |
| Tab-completion misses `init` / `start` / `status` / `migrate` / newer commands | The generated command list is stale/incomplete, so those top-level commands don't tab-complete. Type them in full until the generator is refreshed. |
| Completion also registers `npx @swarmdo/cli@v3alpha` | Targets a dist-tag that does not exist (this fork ships only `latest`). Cosmetic — the `swarmdo` registration is the one that matters. |

The inspectors (`status`, `environments`, `migrate status`, `workflow list`) and all `--help` screens are read-only and safe for hooks/CI. The `deploy` / `rollback` / `release` / `migrate run` / `workflow run` commands change state — gate them behind `--dry-run` and a preceding `status` check.
